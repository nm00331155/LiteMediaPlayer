問題1: ZIP コミック読み込みクラッシュ（ログから原因特定済み）
クラッシュのスタックトレース:

java.lang.IllegalArgumentException: Invalid URI: content://com.android.providers.downloads.documents/document/msf%3A1000001766
    at android.provider.DocumentsContract.getTreeDocumentId(DocumentsContract.java:1294)
    at androidx.documentfile.provider.DocumentFile.fromTreeUri(DocumentFile.java:136)
    at com.example.litemediaplayer.comic.ComicReaderViewModel.resolveDisplayName(ComicReaderViewModel.kt:660)
ZIP コミックの問題はメモリとは一切関係ありませんでした。原因は resolveDisplayName() メソッドです。

ZIPファイルは ActivityResultContracts.OpenDocument で取得された 単一ドキュメント URI（content://com.android.providers.downloads.documents/document/msf:...）です。しかし resolveDisplayName() は最初に DocumentFile.fromTreeUri() を呼んでいます。単一ドキュメント URI に対して fromTreeUri を呼ぶと getTreeDocumentId() が IllegalArgumentException を投げます。

現在のコード:

Copyprivate fun resolveDisplayName(uri: Uri): String {
    return DocumentFile.fromTreeUri(context, uri)?.name           // ← ここでクラッシュ
        ?: DocumentFile.fromSingleUri(context, uri)?.name
        ?: "comic"
}
fromTreeUri は内部で DocumentsContract.getTreeDocumentId() を呼び、ツリー URI でない場合に例外を投げます。?. のnull安全呼び出しは 例外を捕捉しない ので、クラッシュします。

修正（ComicReaderViewModel.kt）:

resolveDisplayName を以下に置換:

Copyprivate fun resolveDisplayName(uri: Uri): String {
    // 単一ドキュメント URI を先に試す（ZIP/CBR は常にこちら）
    try {
        DocumentFile.fromSingleUri(context, uri)?.name?.let { return it }
    } catch (_: Exception) { }

    // ツリー URI（フォルダ）
    try {
        DocumentFile.fromTreeUri(context, uri)?.name?.let { return it }
    } catch (_: Exception) { }

    // フォールバック: ContentResolver で直接取得
    try {
        context.contentResolver.query(uri, arrayOf(android.provider.OpenableColumns.DISPLAY_NAME), null, null, null)?.use { cursor ->
            if (cursor.moveToFirst()) {
                val nameIndex = cursor.getColumnIndex(android.provider.OpenableColumns.DISPLAY_NAME)
                if (nameIndex >= 0) {
                    cursor.getString(nameIndex)?.let { return it }
                }
            }
        }
    } catch (_: Exception) { }

    return "comic"
}
同様に resolveCoverUri と estimatePageCount も fromTreeUri を呼んでおり、ARCHIVE の場合にクラッシュする可能性があるが、現在のコードでは sourceType 分岐で FOLDER のみ fromTreeUri を呼ぶようになっているので問題ない。

問題2: 音量ジェスチャーが正しく動作しない
ログからは直接見えませんが、コードを読むと原因は明確です。

GestureOverlay.kt の onVolumeDelta コールバックに渡される normalizedDelta は、指の移動量を画面高さで正規化した小さな値（例: 0.01〜0.05 程度）です。

PlayerScreen.kt の処理:

CopyonVolumeDelta = { normalizedDelta ->
    val currentVolume = audioManager.getStreamVolume(AudioManager.STREAM_MUSIC)
    val step = (normalizedDelta * maxVolume * 2f).roundToInt()
    val updated = (currentVolume + step).coerceIn(0, maxVolume)
    audioManager.setStreamVolume(AudioManager.STREAM_MUSIC, updated, 0)
    ((updated.toFloat() / maxVolume) * 100f).roundToInt()
},
maxVolume は通常 15 程度。normalizedDelta が 0.02 の場合、step = (0.02 * 15 * 2).roundToInt() = 0。つまり ほぼ毎回 step が 0 に丸められて音量が変わらない。指を大きく動かして離した瞬間だけ delta が大きくなるので、その時だけ 1 ステップ変わる。これが報告されている挙動と完全に一致します。

修正方針: 累積値を使う。小さなdeltaを蓄積し、閾値を超えたらステップを進める。

PlayerPlaybackScreen 内に累積変数を追加し、コールバックを修正:

Copy// 既存の var 宣言の近くに追加
var volumeAccumulator by remember { mutableFloatStateOf(0f) }
var brightnessAccumulator by remember { mutableFloatStateOf(0f) }
音量コールバック:

    registerInteraction()
    volumeAccumulator += normalizedDelta
    val steps = (volumeAccumulator * maxVolume * 1.5f).roundToInt()
    if (steps != 0) {
        val currentVolume = audioManager.getStreamVolume(AudioManager.STREAM_MUSIC)
        val updated = (currentVolume + steps).coerceIn(0, maxVolume)
        audioManager.setStreamVolume(AudioManager.STREAM_MUSIC, updated, 0)
        volumeAccumulator = 0f  // 消費したらリセット
        ((updated.toFloat() / maxVolume) * 100f).roundToInt()
    } else {
        val currentVolume = audioManager.getStreamVolume(AudioManager.STREAM_MUSIC)
        ((currentVolume.toFloat() / maxVolume) * 100f).roundToInt()
    }
},

ログから:
