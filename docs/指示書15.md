LiteMediaPlayer v15 修正指示書
全ソースを確認しました。問題の根本原因を特定し、具体的な修正を以下にまとめます。

確認：ロック機能の仕様
はい、ご認識は合っています。 フォルダ管理画面でロックをオンにすると → そのフォルダは一覧から非表示になる → タイトルを5回タップで認証後に一時的に表示 → 再度5回タップで非表示に戻る。

問題A：プレイヤーのフォルダロックが動作しない
原因（3つ）：

LockManager.isLocked() が常に true を返す問題

toggleLock でロックを新規作成した直後は lastUnlocked に記録がないため、isLocked() は常に true を返す。ここまでは正しい。
しかし buildFolderShells のフィルタ条件は isHidden && isLocked && !showHiddenLocked だが、isLocked の判定に autoLockMinutes が関係している。autoLockMinutes <= 0 の場合は isLocked = false を返す。デフォルト autoLockMinutes = 5 なので初回は true だが、一度アンロックしてから5分以内は false を返す。つまり構造的にはOK。
本当の原因：PlayerScreen の showHiddenFolders がローカル状態のみで、DataStore の hiddenLockContentVisible がアプリ再起動でリセットされる

PlayerScreen の var showHiddenFolders by remember { mutableStateOf(false) } は画面再コンポーズのたびに false にリセットされる。toggleHiddenFolderVisibility(true) で DataStore を更新するが、逆に画面を離れて戻ると showHiddenFolders = false なのに DataStore 側は true のまま。
さらに重大：5回タップで showHiddenFolders = false にした際に toggleHiddenFolderVisibility(false) でDataStoreも更新するが、buildFolderShells の showHiddenLocked はこのDataStoreの値を使っている。つまりフォルダ表示の可否は DataStore の hiddenLockContentVisible で正しく制御されている。
真の原因を発見：PlayerScreen の uiState.folders に表示するフォルダリストの中で、既にフィルタ済みのリストが来ているが、5回タップ後に一度表示されたフォルダをタップして選択しても動画が表示されるかが問題。テスト結果として「フォルダにロックをオンにしても非表示にならない」ということは、フィルタ自体が効いていない。

原因を再確認すると：buildFolderShells で isHidden && isLocked && !showHiddenLocked の条件に isLocked は lockManager.isLocked() を使っている。lockManager.isLocked() は suspend 関数。しかし buildFolderShells 自体は combine の中で呼ばれていて suspend コンテキストだから問題ない。

本当の原因：toggleLock で isHidden = true を設定しているが、lockManager.isLocked() が autoLockMinutes = 5 で lastUnlocked に未記録なので true を返す。 → フィルタは isHidden(true) && isLocked(true) && !showHiddenLocked(false) = true → return@mapNotNull null → フォルダは消えるはず。

しかしユーザーは「消えない」と言っている。ここで lockManager.isLocked を改めて確認：

Copysuspend fun isLocked(targetType: LockTargetType, targetId: Long?): Boolean {
    val config = lockConfigDao.findByTarget(targetType.name, targetId) ?: return false
    if (!config.isEnabled) return false
    if (config.autoLockMinutes <= 0) return false
    val key = createKey(targetType, targetId)
    val lastTime = lastUnlocked[key] ?: return true
    val elapsed = System.currentTimeMillis() - lastTime
    val autoLockMs = config.autoLockMinutes * 60_000L
    return elapsed > autoLockMs
}
targetId は Long? だが、buildFolderShells では lockManager.isLocked(LockTargetType.VIDEO_FOLDER, folder.id) を呼んでいる。folder.id は Long 型（自動採番）。lockConfigDao.findByTarget(targetType.name, targetId) で targetId = folder.id を渡している。

DAO側の問題の可能性： findByTarget がどう定義されているか確認が必要。LockConfig の targetId は Long?（nullable）。DAOで WHERE targetId = :targetId の場合、null 比較は SQL の = NULL になるため false を返す。しかしここでは folder.id は non-null なので問題ない。

最後に可能性として： FolderManagerViewModel.toggleLock が VideoFolder の id を渡すが、PlayerViewModel の buildFolderShells は LockConfig を lockConfigDao.getConfigsByType(LockTargetType.VIDEO_FOLDER.name) で全件取得し、lockConfigs.firstOrNull { it.targetId == folder.id } でマッチしている。ここで型の不一致がないか確認：folder.id は Long、lockConfig.targetId は Long?。Kotlin の == 演算子は Long? と Long の比較で正しく動作する。

結論として、ロジック的にはフィルタは正しく動くはずだが、実際に動作しない原因として最も可能性が高いのは：folderUiFlow の combine が4つの Flow を組み合わせているが、lockConfigDao.getConfigsByType() がリアクティブに更新を発行していない可能性がある（@Query が Flow で LiveData でない場合は即時反映されない場合がある）。

しかしDAOの定義を見ていないので断言はできない。そこで根本対策として、フィルタロジックを確実にする修正を提示します。

2b: シークプレビューの修正
現在のコードの問題点を特定しました：

CopyseekPreviewPositionMs = value.toLong()
// ...
val positionMs = value.toLong()
val positionUs = positionMs * 1_000L
value は sliderValue で、valueRange = 0f..(durationMs.toFloat()) つまりミリ秒単位。positionMs * 1_000L はマイクロ秒変換で正しい。

MediaMetadataRetriever.getFrameAtTime() の第1引数はマイクロ秒（μs）なので、positionMs * 1_000L は正しい変換。

では何故動かないか？

seekRetrieverState の初期化タイミング：remember(videoUri) で1回だけ実行される。runCatching 内で setDataSource を呼ぶが、content:// URI の場合、openFileDescriptor で取得した fd を setDataSource(fd) に渡している。しかし ParcelFileDescriptor は seekRetrieverState のペアの第2要素として保持されているものの、GCで閉じられる可能性がある。

しかし DisposableEffect で onDispose 時にクローズしているので、アクティブな間は保持されているはず。

最も可能性が高い原因：delay(80) のデバウンスが短すぎて、連続ドラッグ中にジョブがキャンセルされ続け、結局一度もフレーム取得が完了しない。ドラッグが止まった瞬間のみフレームが取得されるが、その直後に onValueChangeFinished で seekPreviewBitmap = null にされる。

つまり：

ドラッグ中 → previewJob?.cancel() → 新しい previewJob 開始（delay(80) 待ち） → すぐ次の drag イベントでキャンセル → 以下繰り返し
ドラッグが80ms以上止まる → フレーム取得開始 → しかし getFrameAtTime は重い処理（100ms〜数秒）
ドラッグ再開 → previewJob がキャンセルされる
ドラッグ終了 → onValueChangeFinished で 即座に seekPreviewBitmap = null
これが根本原因です。 フレーム取得が完了する前にジョブがキャンセルされるか、完了してもすぐ null にリセットされる。

修正方法：

CopyonValueChange = { value ->
    registerInteraction()
    isSliderDragging = true
    sliderValue = value
    seekPreviewPositionMs = value.toLong()

    // プレビュー取得をデバウンスするが、キャンセルしない（前回の結果を保持）
    previewJob?.cancel()
    previewJob = previewScope.launch {
        delay(150)  // 150ms のデバウンス（80ms → 150ms）
        val positionMs = value.toLong()
        val positionUs = positionMs * 1_000L
        val bitmap = withContext(Dispatchers.IO) {
            val retriever = seekRetriever ?: return@withContext null
            
            // OPTION_CLOSEST_SYNC が最も高速
            runCatching {
                retriever.getFrameAtTime(
                    positionUs,
                    MediaMetadataRetriever.OPTION_CLOSEST_SYNC
                )
            }.getOrNull()
        }
        // null でない場合のみ更新（前回のプレビューを維持）
        if (bitmap != null) {
            seekPreviewBitmap = bitmap
        }
    }
},
onValueChangeFinished = {
    registerInteraction()
    exoPlayer.seekTo(sliderValue.toLong())
    isSliderDragging = false
    // ★ ここで即座に null にしない。少し遅延させるか、フェードアウトさせる
    previewJob?.cancel()
    previewScope.launch {
        delay(300)  // 300ms 後にプレビューをクリア
        seekPreviewBitmap = null
    }
},
