問題A：タブ切り替え・アプリ再起動で非表示に戻らない
根本原因: hiddenLockContentVisible は DataStore に永続保存されている。一度 true にすると、タブ切り替え・アプリ終了・再起動しても true のまま。リセットする処理がどこにもない。

修正方針: 以下の2箇所でリセットする。

1. タブ切り替え時のリセット — Navigation で PlayerScreen や ComicShelfScreen から離れるとき、DisposableEffect でリセット。

PlayerScreen.kt に追加:

Copy// Scaffold の直前に追加
DisposableEffect(Unit) {
    onDispose {
        // タブを離れる時に非表示に戻す
        viewModel.toggleHiddenFolderVisibility(false)
    }
}
ComicShelfScreen.kt に追加:

CopyDisposableEffect(Unit) {
    onDispose {
        viewModel.toggleHiddenComicVisibility(false)
    }
}
2. ViewModel init でのリセット — アプリ起動時に必ず false にする。

PlayerViewModel.kt の init ブロック先頭に追加:

Copyinit {
    // アプリ起動時に非表示状態にリセット
    viewModelScope.launch {
        appSettingsStore.updateHiddenLockContentVisible(false)
    }
    // ... 既存の init 処理 ...
}
ComicReaderViewModel.kt の init ブロック先頭に追加:

Copyinit {
    viewModelScope.launch {
        appSettingsStore.updateHiddenLockContentVisible(false)
    }
    // ... 既存の init 処理 ...
}
注意: プレイヤーとコミックは同じ hiddenLockContentVisible を共有しているため、片方をリセットすると両方に影響する。これは意図的な動作（セキュリティのため、画面を離れたら非表示に戻す）。

問題B：コミック側でロックしてもフォルダが表示されたまま
根本原因: ComicShelfScreen.kt で uiState.folders をそのまま表示しているが、ComicReaderViewModel のフォルダフィルタは正しく動作している（visibleFolders = folderUis.filterNot { it.isLockEnabled && it.isHidden && !showHidden }）。

問題は toggleComicFolderLock() でロックをオンにした直後、DataStore の hiddenLockContentVisible がまだ true の可能性があること。5回タップで表示 → フォルダ管理でロック → しかし showHidden = true のままなのでフィルタされない。

しかしフォルダ管理画面から戻った後も表示されたままということは、そもそも hiddenLockContentVisible が一度も true にされていない場合でもフォルダが見えるということ。

もう一度フィルタロジックを確認:

Copyval visibleFolders = folderUis.filterNot {
    it.isLockEnabled && it.isHidden && !showHidden
}
toggleComicFolderLock で新規作成時は isHidden = true, isEnabled = true。showHidden = false（デフォルト）。なので filterNot(true && true && true) = filterNot(true) = 除外される。これは正しい。

では何故表示されるか？ lockConfigDao.getConfigsByType(LockTargetType.COMIC_FOLDER.name) が Flow であり、toggleComicFolderLock で upsert した直後に Flow が更新されて combine が再発火するまでにタイムラグがある可能性。しかし Room の Flow は即時反応するはず。

もう一つの可能性: upsert で OnConflictStrategy.REPLACE を使っているが、LockConfig の id が自動生成される。既存レコードが存在しない場合は INSERT になり、targetType = "COMIC_FOLDER" で targetId = folderId のレコードが作成される。getConfigsByType("COMIC_FOLDER") はこれを取得する。

実はもう一つ問題がある: LockConfigDao.findByTarget の SQL:

CopyWHERE targetType = :targetType
  AND ((targetId = :targetId) OR (:targetId IS NULL AND targetId IS NULL))
toggleComicFolderLock(folderId) で folderId は Long（non-null）なので targetId = :targetId のパスが使われる。

upsert の問題を発見: @Insert(onConflict = OnConflictStrategy.REPLACE) を使っているが、LockConfig の Primary Key は id（auto-generate）。つまり upsert は常に INSERT（新しいレコード作成）になる。2回目の toggleComicFolderLock では existing != null で existing.copy(isEnabled = !existing.isEnabled) を upsert するが、existing.id はそのまま保持されるので REPLACE が正しく動作する。

結論: ロジック的には正しいはず。フォルダ管理画面で鍵アイコンを押した後、本棚画面に戻ったときに ComicReaderViewModel が同じインスタンスかどうかが問題。

ComicFolderManagerScreen は viewModel: ComicReaderViewModel = hiltViewModel() を使っているが、Navigation で別の画面なので 別の ViewModel インスタンスが作成される可能性がある。

確認: Hilt の @HiltViewModel は Navigation の backStackEntry に紐づく。ComicShelfScreen は MainTab.COMIC.route、ComicFolderManagerScreen は comic_folder_manager ルート。別の backStackEntry なので別の ViewModel インスタンス。

これが根本原因！ ComicFolderManagerScreen 内の viewModel.toggleComicFolderLock() は ComicFolderManagerScreen 専用の ViewModel インスタンスに対して操作する。DB（Room）には反映されるが、ComicShelfScreen の ViewModel は別インスタンスで、そちらの _uiState は DB の Flow 変更で更新される。

しかし ComicReaderViewModel の init で comicFolderDao.observeAll() と lockConfigDao.getConfigsByType() を combine で監視しているため、DB が変更されれば自動的に反映されるはず。

本当の問題を特定: テストしてみてフォルダが消えない場合、allFolders と folders の両方が同じ uiState に入っており、ComicShelfScreen で uiState.folders を使っている。folders は visibleFolders（フィルタ済み）を設定している。これは正しい。

可能性として残るのは: ComicFolderManagerScreen の viewModel: ComicReaderViewModel = hiltViewModel() が ComicShelfScreen と異なるインスタンスを持つため、toggleComicFolderLock が DB に書き込んでも、ComicShelfScreen 側の ViewModel がまだ combine の再発火を受けていない。しかし Room Flow は DB 変更を全 subscriber に通知するので、これは問題ないはず。

最終的な可能性: フォルダ管理画面と本棚画面が 同時にアクティブ（Navigation の saveState = true で本棚が background に保持されている）の場合、本棚画面の ViewModel は生きているが WhileSubscribed で Flow 購読が停止している可能性。しかし ComicReaderViewModel は MutableStateFlow を使っているので、購読停止の問題はない。

問題C：シークプレビューの仕様変更 — ExoPlayer の seekTo で動画画面自体を変更
仕様変更: MediaMetadataRetriever によるサムネイル取得を廃止し、代わりに ExoPlayer 自体を seekTo でその位置にシークする。これにより動画画面が直接その時間のフレームを表示する。小さなプレビューウィンドウは不要になる。

実装方針:

スライダーのドラッグ中に exoPlayer.seekTo(value.toLong()) を直接呼ぶ
ExoPlayer はキーフレームシークを高速に実行できるため、瞬時に画面が更新される
MediaMetadataRetriever 関連のコードは全削除
プレビューサムネイル表示（160x90のBox）も全削除
Copy// PlayerPlaybackScreen 内の変更

// ★ 削除するもの:
// - seekRetriever, seekRetrieverFd, seekRetrieverInitialized の変数宣言
// - LaunchedEffect(videoUri) の retriever 初期化
// - DisposableEffect(Unit) の retriever クリーンアップ
// - seekPreviewBitmap, seekPreviewRequestId の変数宣言
// - previewJob の変数宣言
// - previewScope の宣言

// ★ Slider の onValueChange を簡素化:
Slider(
    value = sliderValue,
    onValueChange = { value ->
        registerInteraction()
        isSliderDragging = true
        sliderValue = value
        // ★ ExoPlayer を直接シーク（動画画面が即座にそのフレームを表示）
        exoPlayer.seekTo(value.toLong())
    },
    onValueChangeFinished = {
        registerInteraction()
        isSliderDragging = false
        // seekTo は onValueChange で既に呼んでいるので、ここでは不要
        // ただし最終位置を確定するために呼ぶ
        exoPlayer.seekTo(sliderValue.toLong())
    },
    valueRange = 0f..(durationMs.takeIf { it > 0L }?.toFloat() ?: 1f)
)

// ★ プレビューBox（if (isSliderDragging && durationMs > 0L) { ... }）を以下に置き換え:
// シーク中の時間表示のみを表示
if (isSliderDragging && durationMs > 0L) {
    val density = LocalDensity.current
    val fraction = (sliderValue / durationMs.toFloat()).coerceIn(0f, 1f)
    val timeTextWidthDp = 80.dp
    val timeTextWidthPx = with(density) { timeTextWidthDp.toPx() }
    val offsetXPx = (fraction * sliderWidthPx - timeTextWidthPx / 2f)
        .coerceIn(0f, (sliderWidthPx - timeTextWidthPx).coerceAtLeast(0f))

    Box(
        modifier = Modifier
            .offset {
                IntOffset(
                    offsetXPx.toInt(),
                    -with(density) { 36.dp.toPx() }.toInt()
                )
            }
            .background(Color.Black.copy(alpha = 0.8f), RoundedCornerShape(4.dp))
            .padding(horizontal = 8.dp, vertical = 4.dp),
        contentAlignment = Alignment.Center
    ) {
        Text(
            text = formatDuration(sliderValue.toLong()),
            color = Color.White,
            style = MaterialTheme.typography.labelMedium
        )
    }
}
