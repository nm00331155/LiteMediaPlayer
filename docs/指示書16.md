修正2: PlayerScreen.kt — シークプレビューの根本修正
問題: PlayerPlaybackScreen が Navigation で複数回再作成される → seekRetrieverState が破棄と再作成を繰り返す → DisposableEffect で release() 済みの Retriever への参照が残る可能性。

修正: seekRetrieverState の初期化と hasStartedPlayback のガードを強化し、リコンポーズ対策を入れる。また、DisposableEffect(seekRetrieverState) のキーが変わるのは remember(videoUri) で URI が変わった場合のみなので、通常は1回しか生成されないはず。しかしログでは Releasing が3回出ている = seekRetrieverState が3回作り直されている = PlayerPlaybackScreen 自体が3回マウントされている。

根本対策として、MediaMetadataRetriever を remember の外に出して、LaunchedEffect 内で非同期初期化する：

Copy// seekRetriever を State として管理
var seekRetriever by remember { mutableStateOf<MediaMetadataRetriever?>(null) }
var seekRetrieverFd by remember { mutableStateOf<ParcelFileDescriptor?>(null) }

// 非同期で初期化（1回だけ）
LaunchedEffect(videoUri) {
    withContext(Dispatchers.IO) {
        runCatching {
            val uri = Uri.parse(videoUri)
            val retriever = MediaMetadataRetriever()
            val descriptor: ParcelFileDescriptor? = if (uri.scheme == "content") {
                runCatching {
                    context.contentResolver.openFileDescriptor(uri, "r")
                }.getOrNull()
            } else {
                null
            }

            if (descriptor != null) {
                retriever.setDataSource(descriptor.fileDescriptor)
            } else {
                retriever.setDataSource(context, uri)
            }

            // 動作確認
            val testDuration = retriever.extractMetadata(
                MediaMetadataRetriever.METADATA_KEY_DURATION
            )
            AppLogger.d("SeekPreview", "Retriever ready. duration=$testDuration")

            seekRetrieverFd = descriptor
            seekRetriever = retriever
        }.onFailure { error ->
            AppLogger.e("SeekPreview", "Failed to init retriever for $videoUri", error)
        }
    }
}

// クリーンアップ
DisposableEffect(videoUri) {
    onDispose {
        AppLogger.d("SeekPreviewTrace", "Releasing seek preview resources")
        previewJob?.cancel()
        runCatching { seekRetriever?.release() }
        runCatching { seekRetrieverFd?.close() }
        seekRetriever = null
        seekRetrieverFd = null
        seekPreviewBitmap = null
    }
}

さらに、3回リコンポーズ問題の根本対応: LaunchedEffect(Unit) の hasStartedPlayback チェックが3回目でもtrueを返す（rememberが保持されていれば）はずだが、もし Navigation で画面が再作成されている場合は保持されない。これは exoPlayer の remember(context) も影響を受ける。

追加対策: PlayerPlaybackScreen のリコンポーズを減らすため、ViewModel 内のスキャン処理が画面遷移後に再実行されないようにする。init ブロック内の selectedFolderId.collect がフォルダ選択時に scanFolderAsync を呼び出し、これが refresh() → folderUiFlow → uiState 更新 → リコンポーズチェーンを引き起こしている。

PlayerPlaybackScreen では uiState を collectAsStateWithLifecycle で購読しているため、uiState が更新されるたびに再コンポーズが走る。しかし PlayerPlaybackScreen 自体が3回マウントされているのは別問題。

        if (!newValue) {
            // ロックをオフにする → 認証が必要
            val targetActivity = context as? FragmentActivity
            if (targetActivity != null) {
                val executor = ContextCompat.getMainExecutor(targetActivity)
                val biometricPrompt = BiometricPrompt(
                    targetActivity,
                    executor,
                    object : BiometricPrompt.AuthenticationCallback() {
                        override fun onAuthenticationSucceeded(
                            result: BiometricPrompt.AuthenticationResult
                        ) {
                            viewModel.toggleLock(folderState.folder)
                        }
                        override fun onAuthenticationError(
                            errorCode: Int,
                            errString: CharSequence
                        ) {
                            // 認証キャンセル → 何もしない
                        }
                    }
                )
                val promptInfo = BiometricPrompt.PromptInfo.Builder()

                    .setSubtitle("フォルダのロックを解除するには認証が必要です")

                            BiometricManager.Authenticators.BIOMETRIC_WEAK or
                            BiometricManager.Authenticators.DEVICE_CREDENTIAL
                    )
                    .build()
                biometricPrompt.authenticate(promptInfo)
            }
        } else {
            // ロックをオンにする → 認証不要
            viewModel.toggleLock(folderState.folder)
        }
    }

セキュリティ設計の注意点
現在のロック機能は「フォルダの非表示」が目的であり、暗号化やファイルアクセス制限ではない。以下のポイントでセキュリティが担保される：

フォルダ管理画面でロックをオフにする → 生体認証/デバイスクレデンシャルが必要
本棚/プレイヤー画面で非表示フォルダを表示する → 5回タップ + 認証（fiveTapAuthRequired が true の場合）
5回タップで表示後のコンテンツアクセス → 許可（表示されている = 認証済みの状態）
アプリ再起動 → hiddenLockContentVisible が DataStore に保存されるため、明示的に非表示に戻さないと表示され続ける。これはセキュリティ的に問題なので、アプリ起動時に hiddenLockContentVisible = false にリセットすることを推奨
